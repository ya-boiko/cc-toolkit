# git-commands

Git workflow commands for Claude Code — commit, PR description generation, GitLab MR creation, and working directory management.

## Install

This plugin lives in the `paperos` marketplace. Enable it via Claude Code:

```
/plugin install git-commands@paperos
```

Or set it directly in `~/.claude/settings.json`:

```json
"enabledPlugins": {
  "git-commands@paperos": true
}
```

Commands are exposed under the plugin namespace, e.g. `/git-commands:commit`, `/git-commands:mr:create`.

## Workspace Mode

Commands adapt their behavior based on a `.workspace.md` file searched upward from `$PWD` to `$HOME`:

```markdown
---
mode: work
---
```

- **`work`** — enables team conventions: task numbers in commits, Jira links in PRs
- **`personal`** — minimal format, no task numbers required

If the file is not found, defaults to `personal`. Add `.workspace.md` to your global `.gitignore`.

Use `/workspace-init` to create this file interactively.

## Commands

### `/commit`

Creates a git commit.

```
/commit [message] | --no-verify | --amend
```

**What it does:**

1. In `work` mode — asks for the Jira task number
2. Runs pre-commit checks by default: `pnpm lint`, `pnpm build`, `pnpm generate:docs`
3. Checks staged files — if nothing is staged, stages all changes automatically
4. Analyzes the diff to detect if changes should be split into multiple commits
5. Generates a commit message

**Commit format:**

- `work`: `[PRJ-1234] feat: description`
- `personal`: `feat: description`

**Commit types:** `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`

**Options:**

- `--no-verify` — skip pre-commit checks
- `--amend` — amend the last commit

### `/mr:describe`

Generates a Pull Request description from the current branch diff and commits.

```
/mr:describe [target-branch]
```

Default target branch: `master`.

**What it does:**

1. In `work` mode — asks for the Jira task number, Jira URL, and whether it's a continuation PR
2. Reads all commits between current branch and target
3. Analyzes the full diff
4. Generates a PR title and description, saves to `prs/<source>__to__<target>.md`

**Output format (work):**

```markdown
[PRJ-1234] Add Easter egg to the header

https://jira.example.com/browse/PRJ-1234

Adding the snowflake to the header:
- A clickable snowflake that brings up the modal
- The modal contains a mini-game with the cat
```

**Output format (personal):**

```markdown
Add Easter egg to the header

Adding the snowflake to the header:
- A clickable snowflake that brings up the modal
- The modal contains a mini-game with the cat
```

**Continuation PR title:** `[PRJ-1234.2] Fix snowflake animation`

**Example output path:** `prs/feat_PRJ-1234-easter-egg__to__master.md`

### `/mr:create`

Creates a GitLab Merge Request via `glab` using a description file generated by `/mr:describe`.

```
/mr:create
```

**Requirements:** `glab` installed and authenticated.

**What it does:**

1. Detects the current branch
2. Finds matching files in `prs/` (by branch name prefix)
3. If multiple files found — shows a numbered list with target branch and modification date, asks which to use
4. Parses the title (first line) and description (rest) from the chosen file
5. Counts commits ahead of the target branch
6. **If 2+ commits** — generates a squash commit message and appends a `## Squash commit message` section to the description (in a fenced code block) so `/mr:merge` can apply it at merge time
7. Pushes the current branch: `git push -u origin <branch>`
8. Creates the MR: `glab mr create --title "..." --description "..." --target-branch <target>`
9. Returns the MR URL

**Squash commit message** — subject line + optional bullet body:
```
[PRJ-1234] feat: add Easter egg to the header

- clickable snowflake triggers a modal
- modal contains a mini-game with the cat
```

> Note: `glab mr create` does NOT accept `--squash-message` — that flag exists only on `glab mr merge`. The squash message lives in the description until merge time, when `/mr:merge` reads it back.

---

### `/mr:merge`

Squash-merges the open MR for the current branch, applying the squash-message section that `/mr:create` left in the description.

```
/mr:merge
```

**What it does:**

1. Looks up the open MR for the current branch via `glab mr view --output json`
2. Counts commits in the MR via `glab api .../commits`
3. **If 1 commit** — merges without a custom squash message; GitLab keeps that commit's message
4. **If 2+ commits** — extracts the fenced code block under `## Squash commit message` from the MR description
5. If the section is missing — asks: use the MR title, merge without a custom message, or cancel
6. Shows a confirmation summary (MR, target, squash message, commit count) and asks to proceed
7. Runs `glab mr merge <iid> --squash --squash-message "$MSG" --remove-source-branch --yes`
8. Suggests local cleanup: `git checkout <target> && git pull && git branch -D <branch>`

**Typical workflow:**

```
/mr:describe develop   → creates prs/feature-x__to__develop.md
/mr:create             → pushes branch, creates MR from that file
/mr:merge              → squash-merges with the prepared squash message
```

### `/release`

Creates a full release: version bump, changelog, git tag, and publishes to GitHub or GitLab.

```
/release [patch|minor|major]
```

**Requirements:** `gh` (GitHub) or `glab` (GitLab) installed and authenticated.

**What it does:**

1. Detects platform from remote URL (GitHub / GitLab / asks for self-hosted)
2. Reads last tag, proposes patch/minor/major versions, user confirms
3. Generates changelog from commits grouped by Added / Fixed / Other — user can edit before publishing
4. Bumps version in `package.json`, `pyproject.toml`, `Cargo.toml` if found in project root
5. Creates version bump commit (work mode includes task list in body)
6. Creates annotated tag, pushes commit + tag
7. Publishes release with changelog as release notes

**Version bump commit (work mode):**
```
chore: bump version to v1.2.0

[PRJ-1] add Easter egg to header
[PRJ-2] fix snowflake animation
```

**Version bump commit (personal mode):**
```
chore: bump version to v1.2.0
```

### `/workspace-init`

Creates a `.workspace.md` file in the current directory to set the workspace mode.

```
/workspace-init [work|personal]
```

Without an argument — asks interactively. With an argument — creates the file immediately. If `.workspace.md` already exists, asks for confirmation before overwriting.

---

### `/cd`

Switches the working directory for all subsequent operations in the current session.

```
/cd [directory]
```

Without arguments — lists subdirectories and asks which one to use. With an argument — switches immediately.

Useful when working from a parent directory that contains multiple projects (e.g. worktree directories created by `wt`).

**Recommended companion:** Add a SessionStart hook to `~/.claude/settings.json` that auto-detects multi-project directories and asks which one to use:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "dirs=$(find . -maxdepth 1 -mindepth 1 -type d -exec test -e '{}/.git' ';' -print | sort | sed 's|^\\\\./||'); if [ -n \"$dirs\" ] && [ ! -e .git ]; then list=$(echo \"$dirs\" | paste -sd ', ' -); echo \"{\\\"hookSpecificOutput\\\":{\\\"hookEventName\\\":\\\"SessionStart\\\",\\\"additionalContext\\\":\\\"This directory contains multiple projects: $list. Ask the user which project directory to use as the working directory for this session. Once they choose, cd into it for all subsequent work.\\\"}}\"; fi"
          }
        ]
      }
    ]
  }
}
```

The hook fires only when the current directory has no `.git` but contains subdirectories with `.git`. Inside a single project — silent.

## Uninstall

Set `"git-commands@paperos": false` in `~/.claude/settings.json` (or remove the entry).
